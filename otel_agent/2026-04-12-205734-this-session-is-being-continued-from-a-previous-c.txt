● The fix replaces otel_context.attach(captured_ctx) with
  contextvars.copy_context(). Here's why this works:

  Root cause: FastMCP stores the HTTP request in a Starlette ContextVar. When
  @instrumented_tool spawned a new thread, only OTel's context was being copied
  over (otel_context.get_current()). Starlette's request ContextVar lives in a
  completely separate slot — it was not copied, so
  ctx.request_context.request.headers was inaccessible inside the thread.
  _get_parent_ctx caught the exception and returned None, causing
  subtract_operation to start as a new root span with a fresh trace ID.

  The fix: contextvars.copy_context() creates a snapshot of all context
  variables in the current asyncio task — including Starlette's request context
  AND OTel's span context. cv_ctx.run(fn, ...) executes the tool function in
  that copied context, so ctx.request_context.request.headers works normally and
   _get_parent_ctx successfully extracts the traceparent header.

  Restart your MCP servers and run the agent again — you should now see a single
   unified trace.

● rate() shows the per-second rate of change over the window, multiplied by 60
  to get per-minute. It goes to zero when there's no new activity.

  You'll see a non-zero value only while requests are happening within the last 
  5 minutes.

  Here's the timeline:

  ┌───────────────────────────────┬─────────────────────────────────────────┐
  │             Time              │              What you see               │
  ├───────────────────────────────┼─────────────────────────────────────────┤
  │ Just after running agent      │ ~0.4/min (2 transitions / 300s × 60)    │
  ├───────────────────────────────┼─────────────────────────────────────────┤
  │ 5 minutes later with no new   │ 0 — the transitions have aged out of    │
  │ runs                          │ the window                              │
  ├───────────────────────────────┼─────────────────────────────────────────┤
  │ Running it repeatedly every   │ ~2.0/min (2 transitions × 1 run/min)    │
  │ minute                        │                                         │
  └───────────────────────────────┴─────────────────────────────────────────┘

  To see a non-zero value right now:

  Run the agent a few times in quick succession, then immediately query
  Prometheus. The value will be non-zero for 5 minutes and then decay back to
  zero as the window rolls forward.

  If you want a query that always shows something useful regardless of activity,
   use the raw counter instead:

  # Total transitions ever (never goes to zero)
  orchestrator_state_transitions_total

  # Transitions in the last hour (wider window)
  rate(orchestrator_state_transitions_total[1h]) * 60

  The [5m] window is intentionally short — it's designed to show current load,
  not historical totals. That's the correct behavior for a rate metric.

  1. Go to Explore (left sidebar)
  2. Select Loki from the data source dropdown
  3. Use LogQL queries:

  # All logs from the agent service
  {service="multi-agent-calculator"}

  # Logs from a specific trace (copy the Trace ID from Tempo)
  {service="multi-agent-calculator"} | json trace_id="6b819b37a6c15cbc..."

  # Only errors
  {service="multi-agent-calculator"} | level="error"

  # Handoff events
  {service="multi-agent-calculator"} | "Handoff"

  Linking Logs to Traces:

  When you click a log line in Loki, Grafana shows the trace ID in the details
  panel. Click it to jump directly to that trace in Tempo — the logs and trace
  will be linked by span ID.

  The logs are pushed immediately when logger calls fire, and include the
  otelTraceID and otelSpanID automatically via LoggingInstrumentor.

  My doubt is when i develop many agents and many mcp tools  as the project     
  grows. I will be hard for me to write a code for tracing and KPI in the agent 
   code and mcp code itself. I mean it wont be a good practice for development.

● This is the classic cross-cutting concerns problem. Tracing and metrics are
  infrastructure — they shouldn't live in business logic. Here's how to solve it
   properly:

  ---
  The Problem

  Right now the code is tightly coupled:
  # Every tool author must know about OTel
  @add_sub_mcp.tool()
  @instrumented_tool("add", "add_sub_server")   # <-- developer must add this
  def add(a: float, b: float, ctx: Context):
      with tracer.start_as_current_span("add_operation") as span:  # <-- and 
  this
          span.set_attribute("operand_a", a)  # <-- and this

  As you add 50 tools, every developer has to understand OTel. That's wrong.

  ---
  The Solution: 3-Layer Architecture

  ┌──────────────────────────────────────┐
  │  Business Logic (tools, agents)      │  ← developers write ONLY this
  ├──────────────────────────────────────┤
  │  Instrumentation Layer (middleware)  │  ← auto-instruments everything
  ├──────────────────────────────────────┤
  │  OTel SDK (otel_setup.py)            │  ← you already have this
  └──────────────────────────────────────┘

  ---
  Concrete Approach for Your Stack

  1. FastMCP — ASGI Middleware (zero-touch for tool developers)

  Instead of decorating each tool, add one middleware to the FastMCP app. It
  intercepts every HTTP request before it hits any tool:

  # instrumentation/mcp_middleware.py
  class MCPTracingMiddleware:
      """Wraps every FastMCP tool call with a span + metrics. Zero changes to 
  tool code."""
      async def __call__(self, scope, receive, send):
          tool_name = extract_tool_name(scope)  # from the MCP request body
          parent_ctx = otel_extract(dict(scope["headers"]))
          with tracer.start_as_current_span(f"mcp.tool.{tool_name}",
  context=parent_ctx):
              tool_invocations.add(1, {"tool": tool_name})
              await self.app(scope, receive, send)

  # Applied once — covers ALL tools forever
  add_sub_mcp = FastMCP(name="add_sub_server")
  add_sub_mcp.app = MCPTracingMiddleware(add_sub_mcp.app)

  Tool code becomes clean:
  @add_sub_mcp.tool()
  def add(a: float, b: float) -> float:
      """Add two numbers."""
      return a + b   # pure business logic, zero OTel

  ---
  2. OpenAI Agents SDK — Centralized Hook Registry

  Instead of wiring hooks in every agent definition, register them once:

  # instrumentation/agent_hooks.py
  def attach_orchestrator_hooks(agent, worker_name: str):
      """Call once per agent — attaches all KPI tracking."""
      agent.on_handoff = make_on_handoff(worker_name)
      # any future hooks added here automatically apply to all agents

  ---
  3. LangGraph — Node Wrapper

  Wrap nodes at graph construction time, not inside each node:

  # instrumentation/langgraph_hooks.py
  def traced_node(fn):
      """Wraps any LangGraph node with a span + step counter. No OTel in node 
  code."""
      def wrapper(state):
          with tracer.start_as_current_span(fn.__name__) as span:
              result = fn(state)
              step_total.add(1, {"node": fn.__name__, "status": "success"})
              return result
      return wrapper

  # Graph construction:
  StateGraph(MathState)
      .add_node("parse",    traced_node(parse_node))   # one word change
      .add_node("evaluate", traced_node(evaluate_node))

  ---
  4. Central Metrics Registry

  Define all metrics in one file. Nothing else imports OTel meter directly:

  # instrumentation/metrics.py
  from otel_setup import get_meter
  _meter = get_meter("platform")

  # MCP Tool metrics
  tool_invocations = _meter.create_counter("mcp.tool.invocations", ...)
  tool_duration    = _meter.create_histogram("mcp.tool.duration", ...)

  # LangGraph metrics
  step_total         = _meter.create_counter("langgraph.step.total", ...)
  execution_duration = _meter.create_histogram("langgraph.execution.duration",
  ...)

  # Orchestrator metrics
  active_workers     =
  _meter.create_up_down_counter("orchestrator.active.workers", ...)

  Every module imports from here — metrics are defined exactly once.

  ---
  End Result

  ┌─────────────────┬───────────────────────────────────────────────────────┐
  │       Who       │                    What they write                    │
  ├─────────────────┼───────────────────────────────────────────────────────┤
  │ Tool developer  │ Pure Python function, no OTel                         │
  ├─────────────────┼───────────────────────────────────────────────────────┤
  │ Agent developer │ Business logic + attach_orchestrator_hooks(agent,     │
  │                 │ name)                                                 │
  ├─────────────────┼───────────────────────────────────────────────────────┤
  │ Platform        │ Maintains instrumentation/ — one place for all        │
  │ engineer        │ observability                                         │
  └─────────────────┴───────────────────────────────────────────────────────┘

  Adding a new tool gives you full tracing, metrics, and logs automatically with
   zero extra work. Adding a new KPI means editing one file
  (instrumentation/metrics.py) and one middleware — not touching 50 tool files.

  ---
  This is the pattern used at scale — Datadog, Honeycomb, and New Relic's SDKs
  all work this way: instrument the framework, not the application code.
